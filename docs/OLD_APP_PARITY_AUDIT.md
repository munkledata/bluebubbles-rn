## 🔴 High-priority shortlist (all screens)

The genuine, high-value gaps to tackle first, grouped by screen.

**Inbox / conversation list**
- **Mark Read / Mark Unread per-chat action** — Old app's long-press peek menu AND right-click context menu both include a 'Mark Read'/'Mark Unread' toggle (ChatsSvc.setChatHasUnread / toggleHasUnreadAsync). RN's ChatActionsSheet only has Pin/Mute/Archive/Delete — there is no way to manually mark a conversation read or unread from the inbox. The DB layer also has no setChatHasUnread-style repo fn wired into the sheet.
**Chat view (message list)**
- **Copy message text** — Long-press menu 'Copy' copies the message's full text to the clipboard (copyText in text_actions.dart, plus an Android SDK<33 confirmation snackbar). One of the most-used actions. RN's long-press menu (MessageActionsOverlay) has NO copy action at all.
- **Forward message** — Long-press 'Forward' (navigation_actions.forward) opens the chat creator pre-filled with the message text + its attachments to send to another chat. Completely absent in RN.
- **Save / download attachment** — Long-press 'Save' (media_actions.downloadAttachment) saves an image/video/file to the device gallery or downloads folder. RN can view attachments but the message action menu has no Save and there is no save-to-device path from the chat action menu.
**Conversation details**
- **Per-chat Android notification settings (system channel)** — chat_options.dart 'Notification Settings' tile (Android SDK>=30) calls MethodChannelSvc.actions.openConversationNotificationSettings(channelId, displayName) to open the OS per-conversation notification channel screen (custom sound, importance, vibration). RN has only a single in-app Mute switch; no route into the Android per-chat channel.
**New chat / contact picker**
- **Selected-recipient chips (real multi-recipient UI)** — Old app renders each chosen recipient as a removable chip in a horizontal 'To:' row (SelectedContactChip); tapping a search result ADDS a chip and clears the input so you keep adding more, backspace on empty input removes the last chip, tapping a chip removes it. RN instead has one raw comma-separated TextInput, and tapping a suggestion REPLACES the whole input string (setRecipient(c.address)) rather than appending — so building a group is manual/error-prone and there is no visual set of who's selected.
- **Existing-chat detection + inline continue-conversation** — As recipients are selected the old app resolves whether a chat with exactly those participants already exists (by chatIdentifier slug for a single address, else participant-handle matching with phone-suffix fuzzy match against 7-15 digit tails). If found it embeds the live MessagesView right in the creator so you continue the existing thread instead of spawning a duplicate; on send it also does a server GET (method;-;address) lookup before creating. RN always calls createChat blind, risking a duplicate/second thread for a contact you already message.
**Fullscreen media viewer**
- **Fullscreen VIDEO playback in the media viewer** — Old app: FullscreenMediaHolder inspects mimeStart and renders FullscreenVideo (media_kit) with a play/pause overlay, mute toggle, auto-play after 500ms, loop-on-complete restart, tap-to-toggle controls, and a 3s auto-hide timer. In RN, videos never open the /media/ route (VideoPlayer.tsx plays inline with native controls in the bubble), and the /media/[guid].tsx viewer ONLY renders <Image> — so tapping a VIDEO thumbnail from the chat-settings Shared Media gallery (chat-settings/[guid].tsx line 614 calls onOpenMedia for videos) opens a black screen with no playback. Concrete broken path.
- **Swipe between images (carousel across all chat images)** — Old app: FullscreenMediaHolder builds a PageView of ALL image attachments in the chat (or a bounded galleryAttachments list), starting at the tapped one; user swipes left/right to page through them, and the app-bar title shows 'N of M'. RN viewer loads a single attachment by guid with no PageView/FlatList and no adjacent-image navigation.
**Find My**
- **Interactive map with location pins** — The old app's centerpiece is a full FlutterMap (OpenStreetMap tiles) rendering device/friend/item markers as custom pins/avatars. Tapping a list row closes the panel and animates the map to that marker (mapController.move + popup). The RN screen is a plain list only — there is NO map at all; 'Open in Maps' just hands off to an external maps app.
- **Real-time socket location updates** — Controller subscribes to the 'new-findmy-location' socket event and live-updates a friend's marker/row as fresh locations stream in from the server, with status-precedence logic so a better fix replaces a worse one (_handleNewFindMyLocation). RN's store does a one-shot fetch/refresh with no live subscription, so friend locations never update until a manual refresh.
- **Redacted-mode privacy masking** — When settings.redactedMode is on, every device/friend/item name and address is replaced with generic 'Device'/'Item'/'Location' in both list tiles and map popups. The RN screen shows raw names/addresses unconditionally — a privacy-parity gap given this fork's emphasis on the hidePreview/redacted pattern.
**Setup / onboarding**
- **Sync Settings page (configure what to sync)** — A dedicated onboarding page (sync_settings.dart) before sync starts, with: a slider for 'Number of Messages to Sync Per Chat' (0-50, default 25); a 'Sync Active Chats Within' time-filter dropdown (1 week / 1 month / 3 months / 6 months [default] / 1 year / All time); a 'Skip empty chats' switch (default on); a 'Sync group chat icons' switch (default off, with a warning note about sync time); a 'Save sync log to downloads' switch; and a 'Start Sync' button. RN hardcodes sync (100 msgs/chat, no time filter, no skip-empty/group-icon/log toggles) and never shows this page.
- **Dedicated Sync Progress screen during onboarding** — sync_progress.dart shows a full-screen 'Syncing… / Sync complete!' page with a live percentage, a linear progress bar, a scrolling live sync-log console (info vs error color coded), a confetti burst on completion, and 'Restore Backups' + 'Finish' action buttons. RN's connect() jumps straight to /home and syncs silently in the background; sync %/counts are only visible later inside the server-management settings sub-page, with no onboarding progress UI, no live log, and no completion/finish step.
- **In-flow permission requests (contacts + notifications)** — request_permissions.dart is an onboarding page with per-permission cards (Contacts, and Notifications on Android 13+/API33+) showing Granted/Denied chips, Grant Permission buttons, an 'Open Settings' path for permanently-denied, live status refresh on app resume, and a 'Missing Permissions — proceed anyway?' confirm dialog on Next. RN requests contacts lazily during background syncContacts and notification permission during startRealtime, with no onboarding permission screen, status UI, or settings-fallback prompt.
**Settings (root)**
- **Media Settings sub-panel link** — Old 'Application Settings' section row (photo icon) opens AttachmentPanel: auto-download attachments, only-on-WiFi, auto-save, save location/relative path, ask-where-to-save, mute in preview/fullscreen, arrow-key/swipe direction. RN root only has a 'Parallel Downloads' stepper; no Media Settings tile or panel.
- **Notification Settings sub-panel link** — Old 'Application Settings' row (bell icon) opens NotificationPanel: send notifications on chat list, notify for reactions, notification sound, text detection, hide message text, notify when incremental sync complete, plus global/per-chat options. RN has no notification-settings tile or panel on the root (only a Redacted Mode toggle indirectly affects preview hiding).
- **Chat List Settings sub-panel link** — Old 'Application Settings' row (list icon) opens ChatListPanel: connection/sync indicators, message status indicators, filtered chat list, filter unknown senders, unarchive on new message, hide dividers, dense tiles, pin config (rows/per-row/order), swipe actions (left/right), move chat-creator button to header, long-press for camera. RN root has no Chat List Settings tile.
- **Conversation Settings sub-panel link** — Old 'Application Settings' row (chat-bubble icon) opens ConversationPanel: delivery timestamps, chat name as placeholder, avatars in DMs, smart suggestions, replies-to-previous, message options order, sync group icons, store last-read, hide names in reaction details, send/receive sounds + volume, auto-open keyboard, swipe-to-open/close keyboard, hide keyboard on scroll, double-tap message, send with enter, scroll to bottom on send. RN root has no Conversation Settings tile.
**Backup & Restore**
- **Restore from a local file (file picker)** — Old app has 'Restore Local' rows (for both settings and themes) that open the OS file picker (FilePicker.pickFiles, allowedExtensions:['json']), read the chosen .json, confirm 'Restore Settings?/Restore Backup?', and apply. RN restore ONLY accepts pasting the raw backup text into a TextInput — there is no button to pick/open a backup file, so a user who exported a file must open it externally and copy its contents. This is a real usability gap on Android.
**Theming / appearance**
- **App Theme mode (Light / Dark / System)** — OLD: first row is a SettingsOptions<AdaptiveThemeMode> ('App Theme') letting the user pick light / dark / system; drives AdaptiveTheme.setThemeMode and emits 'theme-update'. RN has no light/dark/system switch at all — the active preset itself is inherently dark (OLED Dark / Gator), and there's no follow-OS option. This is the single most-used theming control.
**Notifications settings**
- **Dedicated Notifications settings screen/section** — The old app has a whole 'Notifications' settings page (NotificationPanel) with a GLOBAL OPTIONS tab and a CHAT OPTIONS tab (bottom NavigationBar). RN has NO notifications section anywhere in settings.tsx — the few notification-adjacent controls (Redacted Mode, per-chat Mute) are scattered under Privacy / chat-settings. There is no single place to manage notification behavior.
- **Per-chat mute is not honored (mute type stored but not consumed)** — Old app applies chat.muteType so a muted chat suppresses notifications. In RN, mute_type is written to the DB by the Mute switch / action sheet, but buildMessageIntents (intents.ts) and the notify pipeline never read muteType — so muting a chat currently does nothing to stop its notifications. This is a functional parity gap behind the existing Mute UI.
**Chat-list appearance settings**
- **Entire "Chat List" settings panel** — The old app has a full Settings > Chat List panel (chat_list_panel.dart) grouping Indicators, Filtering, Appearance, Swipe Actions, and Misc options for inbox appearance/behavior. The RN Settings screen (app/(app)/settings.tsx) has THEME/CONTACTS/GENERAL/DOWNLOADS/PRIVACY/SERVER/ABOUT sections and NO chat-list appearance section at all — none of the individual toggles below exist anywhere in the RN app (grep for swipe/dense/hideDivider/syncIndicator/statusIndicator/filteredChat/filterUnknown/unarchiveOnNew/pinRows/cameraFAB/moveChatCreator returns zero hits).
- **Swipe Actions for Conversation Tiles (Material) + configurable Left/Right actions** — Toggle 'swipableConversationTiles' (Android/Material only, gated !kIsWeb && !kIsDesktop && !iOS) enables swipe gestures on inbox tiles; when on, two dropdowns pick the Swipe Right Action and Swipe Left Action from MaterialSwipeAction = {pin, alerts (mute), delete, mark_read, archive}. RN has NO swipe gestures on conversation tiles at all — all chat actions are behind a long-press action sheet only. This is the most directly-named item in the audit prompt and is fully absent.
**Attachment / media settings**
- **Auto-download Attachments toggle** — OLD: SettingsSwitch bound to settings.autoDownload — 'Automatically downloads new attachments from the server and caches them internally.' Master on/off switch for background auto-download. RN hardcodes auto-download (images < 5 MB via shouldAutoDownload) with NO toggle to disable it, and no auto-download of non-image types.
- **Only Auto-download Attachments on WiFi toggle** — OLD: SettingsSwitch bound to settings.onlyWifiDownload — restricts auto-download to WiFi connections to save cellular data. RN has no network-type gating on downloads at all.
**Conversation / message-view settings**
- **Conversation/message-view appearance settings screen (entire panel)** — The old app has a dedicated 'Conversations' settings page (ConversationPanel) under Settings with ~20 toggles/tiles governing message-view appearance, sounds, and gestures. The RN settings.tsx has NO conversation/message-view section at all — none of these keys (showDeliveryTimestamps, recipientAsPlaceholder, alwaysShowAvatars, repliesToPrevious, doubleTapForDetails, sendWithReturn, scrollToBottomOnSend, autoOpenKeyboard, etc.) exist anywhere in src/ or app/. This is the umbrella gap; the individual rows below are its contents.
**Private API feature toggles**
- **Dedicated 'Private API Features' settings screen** — The old app has a whole settings panel (PrivateAPIPanel) reachable from Settings > Advanced. It holds the master enable toggle plus all the sub-toggles below, an explanatory feature list, and a status subtitle ('Private API features are set up / not set up on the server'). RN has no such screen at all — only a read-only info row in the ABOUT section.
- **'Enable Private API Features' master toggle (enablePrivateAPI)** — Master switch that turns on all private-API client behaviors; when off, none of the sub-settings show. Subtitle reports whether the server has the helper bundle set up. In RN this is effectively always-on (behaviors are hardwired) with no client-side master gate beyond the server's own private_api flag.
- **'Send Typing Indicators' toggle (privateSendTypingIndicators)** — Switch controlling whether the app emits typing indicators to other iMessage users. In RN, sendTyping() always fires start-typing/stop-typing when connected with no user control (services/index.ts:491, chat/[guid].tsx:362).
- **'Automatic Mark Read / Send Read Receipts' toggle (privateMarkChatAsRead)** — Switch: automatically marks chats read on the Mac server and sends read receipts to other users when you open a chat. In RN, markRead() fires unconditionally on chat open and from the notification mark-read action (chat/[guid].tsx:114, notifications/actions.ts:43) — no way to disable read receipts.
**Redacted (streamer) mode**
- **Group-chat sender name + avatar are NOT redacted (leak)** — In the old app 'Hide Contact Info' + fake avatars mask contact identity EVERYWHERE, including inside a conversation. In the RN app MessageRow.tsx renders the real sender name (line 83, `msg.senderName`) and a real Avatar with the real name/photo (lines 117-119, `name={msg.senderName ...}` / `uri={msg.senderAvatar}`, no `seed` passed) for received messages in a GROUP chat, with no redacted-mode gating. So opening a group chat under redacted mode reveals every participant's real name and photo above each bubble — a concrete privacy leak the feature is meant to prevent. Triggered by: enable Redacted Mode, open any group conversation.
- **Inline chat attachments are NOT hidden (leak)** — Old app 'Hide Attachments' replaces attachment thumbnails with placeholder boxes inside the conversation. RN MessageBubble.tsx renders attachments unconditionally (`atts.map(... <AttachmentView>)` at line 107) with no `redacted` gate, so photos/videos in the message thread still display at full fidelity under redacted mode. (Note: the chat-settings MEDIA GALLERY at chat-settings/[guid].tsx DOES honor redacted mode, but the main message thread does not.) Triggered by: enable Redacted Mode, open a chat with image/video messages.
**Misc / About / Logging**
- **Disable Battery Optimizations** — Android-only tile in Troubleshoot ('Optimizations' section): checks DisableBatteryOptimization.isAllBatteryOptimizationDisabled on load; tapping launches the OS dialog to exempt the app from battery optimization so it can run in the background/receive pushes reliably; shows a green check when already disabled. Genuinely useful for FCM/background delivery reliability on Android. No RN equivalent.
**Scheduled messages**
- **Dedicated 'New Message' create screen with chat picker** — Old app has a full create panel reached via a + FAB on the list. It includes a 'Chat' tile that opens an action sheet listing ALL chats (with avatars) to pick the destination, plus the message field, schedule options, and a live 'Summary' preview. RN can only create a scheduled message from inside an already-open chat's composer — there is no way to start one from the scheduled list, and no chat selector.
- **Completed / sent / error message history sections** — Old list surfaces a 'Completed Messages' section for one-time messages that already fired: sent rows (green checkmark + 'Sent · <date>') and error rows (red exclamation + the server error text or 'Failed to send'). RN's listAllScheduled filters to status='pending' only, so sent and errored rows are never shown to the user — a scheduled message that failed silently disappears from the UI.
**Message reminders**
- **Preset quick-timeframe picker (1 Hour / 3 Hours / 6 Hours / 1 Day / 1 Week / 1 Month + Custom Date)** — In the old app, creating or rescheduling a reminder opens showTimeframePicker: a dialog listing preset offsets from now (1 Hour, 3 Hours, 6 Hours, 1 Day, 1 Week, 1 Month), each with a time-of-day icon (sun/cloud/moon for hours, calendar/week/month glyphs for longer) and the resolved absolute date shown on the right, plus a 'Custom Date' row that opens the native date picker then time picker. RN instead jumps straight to a bare two-step native date→time picker (pickFutureDateTime) with no presets — every reminder requires manually scrolling a full date+time picker. This is the single biggest UX regression on this screen.
**Cross-cutting (sweep)**
- **Android SEND / SEND_MULTIPLE share-target (receive-into-app)** — Old app registers image/video/text/vcard/audio/application SEND & SEND_MULTIPLE intent-filters on MainActivity, and IntentsService.handleIntent() resolves the shared content URIs to bytes and calls openChat() to prefill either an existing chat or the new-chat creator with the shared text/attachments. Triggered when the user shares a file/text from any other Android app and picks the app. RN app.config has NO android.intentFilters, so the app never appears as a share target and can't receive shared content.
- **Composer camera button (capture-and-send)** — Old iOS-skin composer shows a camera icon: tap opens the system camera for a photo, long-press for video (openFullCamera → ImagePicker.pickImage/pickVideo from ImageSource.camera), staging the capture as a pending attachment. RN composer/AttachmentTray only offers library photos/videos + a Files picker; there is no capture-from-camera entry point in chat (expo-camera is wired only for QR-code setup scanning).
